# Rules

> Wave 1 확정 — 2026-06-01
> 모든 구현 결정은 이 원칙을 우선한다.

---

## R-01. 오프라인 우선 (Offline-First)

**규칙:** 인터넷/셀룰러 없이 모든 핵심 기능이 동작해야 한다.
**이유:** 이 앱의 존재 이유는 인프라가 망가진 상황에서 작동하는 것.
**적용:** 서버 API 호출, 클라우드 저장소, 외부 인증 서비스 사용 금지.

---

## R-02. 중앙 서버 없음 (Decentralized)

**규칙:** 중앙 서버에 의존하는 기능을 만들지 않는다.
**적용:** 모든 데이터는 기기 로컬 저장. 라우팅은 P2P.

---

## R-03. 재난 네트워크 가정

**규칙:** 인터넷, 기지국, 라우터, 일반 LAN은 모두 DOWN으로 가정한다.
**적용:** 외부 서버 호출, DNS, 공유기 경유 통신 코드 금지.
**예외:** 잔존 LAN은 Optional 기능으로만 처리. 핵심 기능 의존 금지.

---

## R-04. 통신 계층 — BLE 단일화

**규칙:** MVP에서 모든 통신은 BLE만 사용한다. WiFi 계층 없음.
**이유:** WiFi Direct(Windows 구현 난이도), 핫스팟 동시 운영 불가 문제를
         PC-to-PC-through-Phone 구조로 해결하여 아키텍처를 단순화.
**적용:**
- `dart:io` Socket, WiFi Direct, WiFi 핫스팟 관련 코드 MVP에서 금지.
- WiFi는 Phase 2 이후 별도 검토.

---

## R-05. BLE 역할 분리

**규칙:** 폰 = Peripheral+Central 동시. PC = Central 전용.
**이유:** `flutter_blue_plus`가 Windows에서 BLE Peripheral 미지원.
**적용:**
- 폰이 항상 Peripheral로 광고하고 연결을 받는다.
- PC는 Central로만 동작하며 폰에게 연결을 요청한다.
- PC끼리는 직접 연결 없음. 폰 메시 네트워크를 통해서만 통신.

---

## R-06. node_id는 공개키 해시

**규칙:** node_id = SHA-256(공개키) 앞 16 bytes. 자가 선언 ID 금지.
**이유:** 임의 UUID를 ID로 쓰면 타인 사칭 가능.
**적용:**
- 앱 설치 시 Ed25519 키쌍 생성, 개인키는 로컬 암호화 저장.
- node_id는 공개키로부터 결정적으로 생성 → 위조 불가.

---

## R-07. 모든 패킷 서명 필수

**규칙:** 모든 패킷(KEY_ANNOUNCE 포함)에 Ed25519 서명을 포함한다.
**이유:** 서명 없으면 발신자 위조 및 가짜 키 주입이 가능하다.
**적용:**
- 서명 검증 실패 패킷: 즉시 폐기, 릴레이하지 않음.
- KEY_ANNOUNCE, ADMIN_NOTICE, TEXT 등 msg_type 무관하게 적용.

---

## R-08. TOFU 공개키 신뢰 모델

**규칙:** 사용자가 핑거프린트를 직접 확인한 경우에만 신뢰 연락처로 등록한다.
**이유:** 오프라인 환경에서 PKI 서버 없이 키 진위를 확인하는 유일한 방법.
**적용:**
- KEY_ANNOUNCE로 수신한 키: 임시 목록 (미확인).
- QR 또는 구두 핑거프린트 대조 후: 신뢰 연락처 승격.
- 기존 신뢰 연락처의 공개키 변경 감지 시: 경고 + 수동 재확인 요구.

---

## R-09. Reverse Path Filtering

**규칙:** 메시지를 받은 BLE 연결로는 같은 msg_id를 재전송하지 않는다.
**이유:** 단순 Flooding의 O(N²) 전송 폭풍 방지.
**적용:**
- msg_id 캐시: 보존 30분, 최대 10,000개 FIFO.
- 릴레이 시 수신 연결 제외, 나머지 연결에만 전달.

---

## R-10. KEY_ANNOUNCE 주기 재전송

**규칙:** KEY_ANNOUNCE는 앱 시작 시 + 이후 5분마다 재브로드캐스트한다.
**이유:** 나중에 범위에 들어온 노드도 공개키를 수신할 수 있어야 함.

---

## R-11. Heartbeat

**규칙:** 30초마다 PING 전송, 2회 연속 무응답 시 이웃 목록에서 제거한다.
**이유:** 죽은 노드로의 반복 전송 방지.

---

## R-12. 배터리 효율 (모바일)

**규칙:** BLE 스캔 기본 모드는 SCAN_MODE_LOW_POWER.
**적용:**
- 연결 요청·메시지 전송 임박 시만 일시적으로 BALANCED 전환.
- 배터리 20% 이하: 광고 주기 자동 연장.
- PC 노드는 전원 연결이므로 적용 제외.

---

## R-13. 메시지 크기 제한

**규칙:** 일반 텍스트 payload는 암호화 전 기준 160자(UTF-8) 이내.
공지S / 공지L은 50자 이내이며, 기본 사용 제한은 1일 1회다.
**이유:** BLE MTU 제약 및 재난 상황에서 핵심 정보만 전달.
**적용:** UI 입력 제한. 공지 cooldown은 로컬 정책과 패킷 검증 양쪽에 적용한다.
음성·이미지는 Phase 4.

---

## R-14. 신뢰할 수 없는 릴레이 환경

**규칙:** 릴레이 노드가 악의적일 수 있다고 가정하고 설계한다.
**적용:**
- payload는 수신자 공개키(X25519+AES-GCM)로 암호화.
- 릴레이 노드는 암호화된 payload 열람 불가.
- 서명 검증 실패 패킷은 릴레이하지 않음.
